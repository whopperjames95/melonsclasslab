Classes: Melons

Classes

Classes are a useful way to model the combination of data and logic.

We might make a salesperson class to model information about salespeople like so:

class Salesperson:
    """A salesperson at Ubermelon."""

    base_salary = 50000
    commission_rate = 0.05

    def calculate_monthly_pay(self, total_of_sales):
        """Determines amount of monthly pay."""

        return self.base_salary / 12 + (total_of_sales * self.commission_rate)

We could then make a salesperson like this:

>>> jayda = Salesperson()

And we can call the calculate_monthly_pay method on her like this:

>>> jayda.calculate_monthly_pay(total_of_sales=1000)
4216.0

Of course, not all salespeople get paid the same. We could have a more senior saleswoman, Juanita, who gets a higher commission:

>>> juanita = Salesperson()
>>> juanita.commission_rate = 0.10

Each “instance” of a class (often called an “object”) can have different attributes (like base_salary and commission_rate).

self

So, what’s that “self” in our class?

That’s the actual individual salesperson! In calculate_monthly_pay, when we talk about self.base_salary, we don’t mean “the base salary that all salespeople have” (since that can change from salesperson to salesperson!); we mean the base salary of this individual salesperson.

Functions on classes (called “methods”) always take self as their first parameter. You don’t pass this to the method explicitly—instead, when you call the method on an instance of the class, it’s passed automatically.

Huh? What’s that mean?

In other words, since our calculate_monthly_pay has self as it’s first argument, when we call it like:

>>> juanita.calculate_monthly_pay(5000)

The argument for self becomes Juanita herself! The 5000 we pass becomes the second argument, total_of_sales.

__init__

It’s a bit of a pain to have to create a salesperson and then manually change their commission rate and base pay, though – it would be nice if we could set this when we create them.

If a class defines a __init__ method, this is automatically called when we create instances (“instantiate objects”) from the class.

We could update our Salesperson class to have this:

class Salesperson:
    """A salesperson at Ubermelon."""

    def __init__(self, base_salary=50000, commission_rate=0.05):
        self.base_salary = base_salary
        self.commission_rate = commission_rate

    def calculate_monthly_pay(self, total_of_sales):
        """Determines amount of monthly pay."""

        return self.base_salary / 12 + (total_of_sales * self.commission_rate)

Note that __init__ doesn’t return anything–it just updates the newly-made instance to change things about it.

Our salespeople use the following rules to determine the price of melons (rules are applied in this order):

  • Normally, the base cost of melons is $5/each.

  • The base cost of Casabas and Ogens is $1/each more, though, as they’re harder to grow.

  • Imported melons cost 1.5 times as much as they otherwise would, because of shipping cost.

  • Square melons cost 2 times as much as they otherwise would, because of the much more intense effort to grow them.

Right now, we’ve just had our salespeople look at the data file and figure out how much a melon will cost by applying all of these rules to the melon.

So, for example:

  • a Horned Melon (which is foreign-grown, natural-shaped) would cost $5 × 1.5 (for being foreign-grown) = $7.50

  • A Casaba is $9 (base price $5 plus $1 for being a Casba or Ogen, times 1.5 for being imported).

We have some new challenges, though: some of our melons are discounted for multiple purchases. Starting now, we’re offering the following discounts:

  • If you buy 3 or more Watermelons, you get all of them at 75% of what the cost would otherwise be. So, determine the price for all of the Watermelons someone wants and, if it’s three melons or more, multiply that total by 0.75.

  • If you buy 5 or more Cantaloupes, you get them all at half price (because, really, who likes Cantaloupe that much?).

This is really a pain for our salespeople, and also makes it hard to get data right for our web site.

As we’ve been learning about classes, this seems like a great time to use classes—since we’re combining data and functionality.

def get_price(qty):

Part II: Hierarchies

First, Some More Learning

We can have one class “inherit” from another class, where it gets all of the attributes (data) and methods (smarts) from a “parent” class.

For example:

class InternSalesperson(Salesperson):
    """A college intern at Ubermelon working in sales."""

Since InternSalesperson “inherits” from Salesperson, everything a salesperson can do, an InternSalesperson can do:

>>> jessica = InternSalesperson()
>>> jessica.base_salary
50000

Oops. That’s a lot to pay an intern.

So we could modify the InternSalesperson class to “override” some attributes or methods:

class InternSalesperson(Salesperson):
    """A college intern at Ubermelon working in sales."""

    def __init__(self, base_salary=20000, commission_rate=0.05):
        self.base_salary = base_salary
        self.commission_rate = commission_rate

(Note the change from 50k to 20k)

Now, InternSalesperson objects will run a slightly different __init__ method when they’re created, but they still share the same calculate_monthly_pay method.

Now, Back To Your Exercise

Our CEO is always toying with changing the base price of melons from $5 to something else, sometimes on a whimsical basis.

Right now, you probably have code that looks like this:

class WatermelonOrder:
    species = "Watermelon"
    color = "green"
    imported = False
    shape = 'natural'
    seasons = ['Fall', 'Summer']

    def get_price(self, qty):
        total = 5.0 * qty
        if qty >= 3:
            total = total * 0.75
        return total

(and similar classes for all of the other melon order types)

It would be a pain to change the base $5.00 cost of melons for each of the melons by changing each class individually, though.

You could make the base price come from a constant, like:

BASE_MELON_PRICE = 5.00

class WatermelonOrder:
    species = "Watermelon"
    color = "green"
    imported = False
    shape = 'natural'
    seasons = ['Fall', 'Summer']

    def get_price(self, qty):
        total = BASE_MELON_PRICE * qty
        if qty >= 3:
            total = total * 0.75
        return total

That would make it easier to update–but it wouldn’t be flexible enough for our CEO. Sometimes the CEO talks about making the base price vary on dynamic things, like the weather that day or the day of the week, or other things. If we had to put that dynamic logic into each melon type class, we’d be duplicating a lot of code–so even though it’s $5 for now, we want to plan ahead.

We could solve this by making all of our melons subclass a common base class (MelonOrder might be a good name for this!), and they could get the base price of melons by calling a method, get_base_price() on the parent MelonOrder class. For now, this get_base_price could just return 5—but we’re creating a “hook” where we could later do something smarter/more complex.

That is, if you change the base price on the MelonOrder class, all of your other classes would still be able to get the newly-updated cost.
